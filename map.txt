# PAZATOR Codebase Map

## Project Overview

Pazator is a single‚Äëpage browser‚Äëbased intelligence analysis application (~50K lines: 34K JS, 12.8K CSS, 2.9K HTML). Dark theme, local‚Äëfirst storage but expanded with PZLS allowing local lan networks, AI assistant (Zor) with pluggable provider system (Gemini, DeepSeek), TIDE analysis engine, classification system, logbook, GIS overlays, MapLibre globe tracker (LCTX) with Supabase made to be a input for trackers, entity relationship graph, case management, ontology designer, pipeline ETL, rule engine (alerts + workflow), in-browser API console with GraphQL, report builder, plugin system, and D3 force-directed graph explorer, Karline a way to turn plans into actions.

**Size:** 39 JS modules, ~80 exported functions, 14 tab views, 18 modal dialogs, 2.9K lines of HTML.

**Fonts:** Alliance No.2 (branding), Maven Pro (UI), DM Mono (code/monospace in mobile), Anton (mobile headers).

## Key Architecture Patterns

### IIFE Module Pattern
Every JS module uses an IIFE that exposes a single global on `window`:
- `window.pazatorDashboard`, `pazatorAlerts`, `pazatorAPI`, `pazatorWorkflow`, `pazatorReports`, `pazatorPipelines`, `pazatorOntologyDesigner`, `pazatorObjectExplorer`, `pazatorObjects`, `pazatorLogbook`, `pazatorGIS`, `pazatorRelationships`, `pazatorClassification`, `pazatorStore`, `pazatorFacets`, `pazatorSync`, `pazatorCollab` (referenced in reports.js), `pazatorHumanTile`, `pazatorPlugins`, `pazatorExplorer`, `pazatorAI`, `pazatorAdmin`, `pazatorFile`, `pazatorServerSearchResult`.

### Global State (`js/core/globals.js`)
- `pazatorData = { humans: [], others: [], chats: [] }` ‚Äî main entity arrays
- `tags = []` ‚Äî global tag list
- `cases = []` ‚Äî case objects
- `aiChatHistory = []` ‚Äî Zor conversation history
- `autoSaveInterval`, `pendingChanges`, `selectedCaseId`

### Event Bus (`Tastur`)
Custom global event bus (`Tastur`). Defined in `js/core/plugins.js` if not already present by plugin registration time. Used by persistence (`data_saved`, `data_loaded`), intel (`threat_detected`), search (`search_performed`), plugins (lifecycle hooks), etc. Supports `on()`, `off()`, `emit()`.

### Lazy Module Loading (`js/ui/tabs.js` ‚Üí `ensureLazyModule()`)
Dynamic `<script>` injection on first tab click. Modules: dashboard, ontology, pipelines, alerts, api, workflow, reports, explorer, sync, walkthrough, snappy, gis, lsad.

## Data Model

### Humans (`store._data.humans`)
~30 fields: `id`, `name`, `objectType` (default `'Person'`), `gender`, `birthDate`, `workplace`, `credit` (0-370, default 185), `socialClass`, `friends[]`, `family[]`, `extraNotes`, `tags[]`, `imagePreview` (base64), `chats[]`, `cases[]`, `trackerAlias`, `trackerLinkedAt`, `maritalStatus`, `nationality`, `countryOfOrigin`, `immigrationStatus`, `languages`, `ethnicity`, `religion`, `politicalViews`, `threatLevel` (None/Low/Medium/High/Critical), `incomeLevel`, `educationLevel`, `classification` (optional `{level, classifiedBy, classifiedAt}`).

### Others (`store._data.others`)
Simpler: `id`, `name`, `objectType` (default `'Organization'`, can be Vehicle/Location/Event/etc.), `type`, `note`/`notes`, `imagePreview`, `classification`.

### Relationship Edges (`pazatorRelationships`)
Flat array: `{id, sourceId, targetId, sourceType, targetType, type (13 types), strength (1-5), confidence (0-1), notes, createdAt, updatedAt, metadata}`. O(n) scan for all queries. Dual system with legacy `friends`/`family` inline arrays on humans ‚Äî `getRelatedObjects()` merges both with no dedup.

### Cases
`{id, title, description, status (open/in-progress/closed), entities: [{id, name, type}], evidence: [{id, type, title, findingCount, addedAt}], timeline: [{type, content, timestamp}], createdAt}`. Stored in `pazatorCases` localStorage key + IndexedDB KV.

### Logbook Entries
`{id, title, body, category (general/observation/incident/intel/comm), createdAt, updatedAt, classification}`. Stored in `pazator_logbook` localStorage key.

### Chat Objects
`{id, source (whatsapp/telegram/discord/signal/manual), participants[], content, timestamp, context, parsed}`. Managed by `ChatStorageManager` with IndexedDB KV + localStorage `chatHistory` dual-write, TTL cache, archive system.

### Pipeline Objects
`{id, name, connector: {type, config}, transforms: [{type, config}], targetStore, createdAt, updatedAt}`. Stored in `pazator_pipelines` localStorage. Run history capped at 200 in `pazator_pipeline_runs`.

### Persistence Flow

```
User action ‚Üí pazatorData.humans.push(x)
  (pazatorData.humans IS store._data.humans ‚Äî same proxied array reference)
  ‚Üí proxy traps push/splice/set
  ‚Üí 1. invalidateCache()
  ‚Üí 2. rebuildIndexes() (_objectIndex, _humanIndex, _nameIndex)
  ‚Üí 3. store.markDirty(storeName)
  ‚Üí 4. emit('entity_changed', {id, type, action})
  ‚Üí 5. emit(key + '_changed', ...)
  ‚Üí 2s debounce flushDirty() ‚Üí syncToEngine() ‚Üí IndexedDB putMany()
```

Globals and store share the same proxied arrays ‚Äî no sync step between them. `markDataChanged()` no longer copies data (was already broken, writing to a spread copy). `saveData()` kept as a 30s periodic safety net but only calls `syncToEngine()`.

No localStorage involved in data persistence. All entity data flows through in-memory store ‚Üí IndexedDB v2 (`PazatorV2`). The legacy `pazatorDB` (IndexedDB v1) and migration functions have been removed.

## Sync Protocol (`js/apps/sync.js`)

**PZLS server endpoints (all at `http://<host>:3456`):**

Auth (rate-limited, 10/min by default):
- `POST /api/auth/register` ‚Äî create account `{username, password, email?, displayName?}`
- `POST /api/auth/login` ‚Äî login ‚Üí `{token, user}`
- `POST /api/auth/logout` ‚Äî invalidate session
- `GET /api/auth/me` ‚Äî current user info (includes email, displayName)
- `POST /api/auth/change-password` ‚Äî `{currentPassword, newPassword}`
- `POST /api/auth/update-profile` ‚Äî update email/displayName
- `GET /api/auth/sessions` ‚Äî list your active sessions
- `DELETE /api/auth/sessions/:token` ‚Äî kill your session

Sync:
- `GET /api/sync` ‚Äî full pull (all entities)
- `POST /api/sync` ‚Äî full push `{clientVersion, stores}`
- `GET /api/sync/cursor` ‚Äî **incremental pull** (cursor-based pagination, `?cursor=&limit=&store=`)
- `GET /api/sync/ops?since=N&limit=500` ‚Äî operation log (paginated)
- `POST /api/sync/ops` ‚Äî push individual operations (with version-based conflict detection)
- `GET /api/sync/changes?since=N` ‚Äî changes since version
- `GET /api/sync/status` ‚Äî server health + account stats
- `GET /api/sync/stores` ‚Äî record counts per store
- `POST /api/sync/encryption-key` ‚Äî set E2E encryption key
- `GET /api/sync/encryption-status` ‚Äî check if encryption key is set

Search:
- `GET /api/search?q=&store=&page=&perPage=` ‚Äî server-side SQL LIKE search across all entity data

Files:
- `POST /api/files/upload` ‚Äî upload file (base64, max 10MB)
- `GET /api/files/:id` ‚Äî download raw file
- `GET /api/files/meta/:id` ‚Äî file metadata
- `GET /api/files?entityId=&entityStore=` ‚Äî list files for entity
- `DELETE /api/files/:id` ‚Äî delete file

Admin (admin role required):
- `GET /api/admin/config` ‚Äî runtime info, paths, env vars
- `GET /api/admin/export` ‚Äî full DB dump as JSON
- `POST /api/admin/import` ‚Äî restore from dump
- `GET /api/admin/users` ‚Äî list all users (with email, display_name, role)
- `PATCH /api/admin/users/:id` ‚Äî update user `{disabled, role, email, displayName}`
- `POST /api/admin/users/:id/reset-password` ‚Äî admin password reset (terminates sessions)
- `DELETE /api/admin/users/:id` ‚Äî delete user
- `GET /api/admin/users/:id/sessions` ‚Äî list user's sessions
- `DELETE /api/admin/sessions/:token` ‚Äî force-kill session
- `GET /api/admin/ops?since=N&limit=200` ‚Äî audit log
- `GET /api/admin/admins` ‚Äî list forced admin usernames
- `POST /api/admin/admins` ‚Äî set forced admin usernames

Other:
- `GET /api/resolve?threshold=0.6` ‚Äî Jaro-Winkler entity resolution
- `POST /api/query` ‚Äî external SQL query `{connectionString, query}`
- `GET /api/auth/health` ‚Äî health check

**Sync model:** Full-state push/replace, full-state pull/append. Also supports cursor-based incremental pull (`GET /api/sync/cursor`) for large datasets. Global integer version counter per entity for conflict detection. Op log for audit trail.

**Conflict resolution:** Version-based optimistic concurrency. When `expectedVersion` mismatches server version, server returns `{conflict: true, serverVersion, serverData}`. Conflicts stored locally in localStorage (`pazator_merge_conflicts`). Resolved via UI (System ‚Üí Resolve Conflicts) with AI auto-solve using configured AI provider.

**Offline queue:** When server is unreachable, push operations are queued in localStorage (`pazator_sync_offline_queue`). Retried on demand via System ‚Üí Flush Queue.

**E2E Encryption:** Client-side toggle in Settings ‚Üí Security ‚Üí E2E Encryption. Data encrypted with AES-256-GCM before leaving browser using PBKDF2-derived key from user passphrase. Encrypted data stored as `{__enc: true, __v: 1, data: "<base64>"}`. Decrypted on pull. Server never sees plaintext.

**Storage (PZLS):** Shared SQLite database (`better-sqlite3`) in `data/pazator.db`. Single `entities` table with `(id, store)` PK, shared across all users. Users table includes email and display_name fields. AES-256-GCM encryption-at-rest for classified entities. Per-user E2E encryption keys stored in `kv` store. Files stored as BLOBs in `files` table. Op log in `ops` table (auto-increment seq). Sessions in `sessions` table with TTL.

**Data synced:** 6 stores ‚Üí object maps keyed by ID: `humans`, `others`, `tags`, `cases`, `chats`, `relationships`.

**Desktop:** `npm run desktop` launches **LSAD** (Electron app wrapping the admin panel). Opens in a native window with menu bar (PZLS, Edit, View, Window).

## Graph & Relationships

### Entity Graph Explorer (`js/graph/explorer.js`)
Dedicated graph tab with full entity relationship visualization. Uses D3 `forceSimulation` with `forceLink`, `forceManyBody(-200)`, `forceCenter`, `forceCollide(28)`.

**Layout:** Workspace with main-panel (full-bleed SVG) and side-panel (controls + info). Sidebar sections: Add Entity (search), Controls (Load All / Reset View / Clear Graph), Stats (node/edge counts), Node Info (expand/details/remove).

**Features:**
- Auto-loads all entities from store on first open
- Search by name to add individual entities (with auto-expand of relations via double-click)
- O(1) edge dedup via `edgeSet` (instead of O(n) array scan)
- Grayscale color palette ‚Äî node shade by entity type, stroke by threat level
- ResizeObserver coalesced to rAF
- Visibility API freezes simulation when tab hidden
- Labels hidden at zoom < 0.3x
- Null-safe tick guards against stale D3 references

**Limitations (current):** No BFS pathfinding UI, no edge filtering, single-select only.

### Other Graph Features
- D3 force-directed graph in `chat-manager.js` ‚Äî renders connections from `deployConnectionAgents()` or shared-attribute heuristics. Supports zoom/pan, tooltips, click-to-open-detail, drag, path highlighting (green). ResizeObserver for responsive re-center.
- BFS pathfinding (`chat-manager.js:findPath()`) ‚Äî shortest path over `pazatorRelationships`, returns `{nodeId, relType}` steps, renders on D3 graph.
- Family graph in `detail.js` ‚Äî circular CSS-based layout for direct family/friend connections only.

## Key Performance Optimizations

| Problem | Solution |
|---------|----------|
| Slow pagination | `cursor.advance()` + virtual lists (chat list + data table) |
| Heavy layout on main thread | Worker-based force layout (`graph_tick`) |
| Excessive re-renders | `entity_changed` events + aggregate cache + `scheduleRender()` coalescing |
| Synchronous storage bottleneck | Dual-write: localStorage sync reads, IndexedDB async persistence |
| 12 blocking scripts | Lazy-load on first interaction (`ensureLazyModule()`) |
| Unthrottled resize | `requestAnimationFrame` throttling (`visualization.js:314`) |
| Redundant AI calls | Hash-keyed cache in `AIQueue` |
| Chat system performance | TTL cache + archive system in `ChatStorageManager` |
| Data loading on startup | `skeletonGraph` CSS loading skeleton while data loads |

## Notable Algorithms & Data Structures

1. **Virtual List** (`chat-manager.js`): `PazatorUI.VirtualList` with itemHeight 130, overscan 3, pagination 25 per page. Fallback DOM rendering.
2. **D3 Force Graph** (`chat-manager.js`): `forceSimulation` with `forceLink`, `forceManyBody(-200)`, `forceCenter`, `forceCollide`. Shared-attribute inferred links (workplace, nationality, etc.). Custom tooltip + drag + zoom/pan.
3. **Multi-Agent Connection Finder** (`chat-manager.js`): Divides humans into N chunks, recursive depth-limited exploration up to `depth` (default 2), 8-attribute comparison for shared values.
4. **BFS Pathfinding** (`chat-manager.js`): Queue-based, visited Set, over `pazatorRelationships` edges.
5. **Entity Timeline Assembly** (`detail.js`): Merges 6 event types (relationships, cases, chats, tracker, analysis, notes) with dedup, sorts desc by timestamp.
6. **Case Related-Entity Expansion** (`cases.js`): Transitive closure one-level: friends + family + workplace from current case entities.
7. **Cross-Entity Search** (`detail.js`): 3 data sources (humans, chats, cases), multi-field matching, 50-result limit, 300ms debounce.
8. **Facet Filtering** (`search.js`, `facets.js`): AND-combined facet filters, OR within a facet. 17 indexed dimensions.
9. **Aggregate Cache** (`store.js`): Lazy-recomputed on mutation, stores threat counts, credit risk, case statuses.
10. **O(n¬≤) Force Layout** (`worker.js`): N-body repulsion + attraction with linear cooling schedule.
11. **Credit Risk Categorization**: Three-tier bins (0-124 high, 125-249 medium, 250-370 low).
12. **Time-Ago Formatting** (multiple files): "just now", "Xm ago", "Xh ago", "Xd ago", date ‚Äî millisecond-diff buckets.
