# PAZATOR Codebase Map


---

## Project Overview

Pazator is a single‑page intelligence analysis application with a dark‑theme UI, local‑first storage (IndexedDB + localStorage), an AI assistant (Zor), a TIDE analysis engine, classification system, logbook, GIS tracking, and LAN sync via a Node.js server (PZLS).

**Companion server:** `https://github.com/AIX-32/PZLS` — SQLite + WebSocket server for multi‑user sync, encryption at rest for classified entities, auth with token expiry.

## Context Selection Modal (`zor-context.js`)

Opened from chat options menu → "Select Context". Asymmetric layout with 7 data source sections:

| Section | Data Source | Loader Function |
|---------|-------------|-----------------|
| People | `pazatorData.humans` | `loadContextPeople()` |
| Chats | `ChatStorageManager.getChatHistory()` | `loadContextChats()` |
| Intel Findings | `fraudLogs` + `terroristLogs` + `previousThreats` + `previousFraud` (localStorage) | `loadContextIntel()` |
| Logbook | `pazatorLogbook.getEntries()` (`pazator_logbook`) | `loadContextLogbook()` |
| Cases | global `cases` array | `loadContextCases()` |
| Entities | `pazatorData.others` | `loadContextEntities()` |
| Reports | `pazator_analysis_reports` (localStorage) | `loadContextReports()` |

Selected items are stored in `localStorage.adminProvidedContext` with a 5‑minute TTL, displayed as pill cards in `#contextDisplay` above the chat input.

## TIDE Engine (June 2026)

**Threat Intelligence Distributed Evaluation** – replaces monolithic AI calls with scalable chunked analysis.

- Handles any dataset size via scoring + tiered chunking
- Analysis types: `threat`, `terrorist`, `fraud`, `credit`, `risk`, `intel`, `connection`
- Dynamic chunk sizing based on token estimation
- Built‑in deduplication, retry, progress events, persistence
- CTXOD integration: `prioritize()` returns highest‑value entities
- Findings persist to localStorage keys: `previousThreats`, `previousFraud`
- Reports persist to `pazator_analysis_reports`

## Data Pipeline

### Storage Layers (bottom → top)

| Layer | File | Description |
|-------|------|-------------|
| **IndexedDB (primary)** | `engine.js` | 7 stores + `_kvStore`. Queries: `getPage()` (cursor‑advance fast path), index‑based (`getAllByIndex`, `countByIndex`, `queryRange`) |
| **localStorage (fallback)** | – | Used when store unavailable at startup. `migrateToIDB()` copies `pazator_*` keys at init |
| **In‑memory caches** | `store.js` | `_aggregateCache` for precomputed stats, `_messageBus` for `entity_changed` events |
| **Chat system cache** | `chat-system.js` | `_cache` with TTL, `_archiveCache`; dual‑writes to IndexedDB `_kvStore` |

### Key localStorage Keys

| Key | Contents | Used By |
|-----|----------|---------|
| `pazatorData` | Full app state JSON (humans, others, chats) | `persistence.js` |
| `pazatorData_backup` | Previous snapshot before save | `persistence.js` |
| `chatHistory` | Chat objects array | `ChatStorageManager`, `zor-context.js` |
| `pazator_logbook` | Logbook entries array | `logbook.js` |
| `pazatorCases` | Cases array | `cases.js` |
| `pazator_analysis_reports` | TIDE report objects `{reports: [...]}` | `report-manager.js`, `findings.js` |
| `previousThreats` / `previousFraud` | TIDE finding arrays | `findings.js`, `zor-context.js` |
| `fraudLogs` / `terroristLogs` | Legacy security log arrays | `chat-manager.js`, `zor-context.js` |
| `adminProvidedContext` | Active AI context `{summary, data}` | `zor-context.js` |
| `pazator_sync_config` / `pazator_sync_state` | LAN sync config + state | `sync.js` |
| `pazator_auth_token` / `pazator_user_cache` | Auth token + cached user | `sync.js` |
| `gemini_api_key` / `gemini_model` | AI provider config | `gemini.js`, `settings.js` |
| `pazator_saved_searches` | Saved facet searches | `facets.js` |

### Data Flow

```
User action → store mutation → invalidateCache() + entity_changed event
            → persistence.scheduleRender() (coalesced DOM update)
            → visualization (only affected entities, no full re‑render)
```

### AI Pipeline

```
js/ai/gemini.js → js/ai/ai.js (AIQueue + caching)
                → js/ai/agents.js (orchestrator)
                → js/ai/context.js (CTXOD tools)
                → js/ai/zor-chat.js (chat UI)
```

**Credit scoring cache:** `cacheKey = hash(person data + length)` – checks `AIQueue.getCached()` before Gemini call.

## Key Performance Optimizations Summary

| Problem | Solution |
|---------|----------|
| Slow pagination | `cursor.advance()` + virtual lists |
| Heavy layout on main thread | Worker‑based force layout |
| Excessive re‑renders | `entity_changed` events + aggregate cache + `scheduleRender()` coalescing |
| Synchronous storage bottleneck | Dual‑write: localStorage for synchronous reads, IndexedDB for async persistence |
| 12 blocking scripts | Lazy‑load on first interaction |
| Unthrottled resize | `requestAnimationFrame` throttling |
| Redundant AI calls | Hash‑keyed cache in `AIQueue` |
